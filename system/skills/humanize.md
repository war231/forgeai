# Workflow: Humanize（进化式去AI味）

> **核心理念**：不是"对照规则删词"，而是"生成多个候选 → 打分 → 更像人的留下 → 一轮轮优化"。
> **灵感来源**：TianyiDataScience/humanize 的 baseline-challenge-score 机制。
> **增强设计**：融入武行网文宪法 + Wikipedia 24种AI特征，双引擎驱动。

---

## 🧬 机制说明

```
输入原文（baseline）
    │
    ▼
┌─────────────────────────────┐
│  Round 1: 生成挑战者         │
│  ├── challenger-A: 启发式自然版  │
│  ├── challenger-B: 启发式平衡版  │
│  └── challenger-C: 规则驱动版    │  ← 武行宪法+24种AI特征
│                              │
│  打分（双引擎）               │
│  ├── 引擎1: 网文AI味检测       │  ← 武行宪法（神态微动作/滥用比喻/翻译腔）
│  ├── 引擎2: 通用AI味检测       │  ← 24种AI特征（三段式/填充短语/谄媚语气）
│  └── 综合评分 = 网文分×0.6 + 通用分×0.4
│                              │
│  更像人 → 保留为 new_baseline │
│  更像AI → 淘汰               │
└─────────────────────────────┘
    │
    ▼  (如果未通过质量门限)
┌─────────────────────────────┐
│  Round 2: 基于最佳候选修复     │
│  ├── 检测残留AI味标签          │
│  ├── 针对性修复                │
│  └── 再次打分                  │
└─────────────────────────────┘
    │
    ▼  (最多3轮)
  输出最终版本 + 优化过程报告
```

---

## 📋 Step 0: 输入确认

**输入**：
- 待优化的正文段落（可以是单章或片段）
- 上下文信息（角色当前状态、场景、情绪基调）

**动作**：
1. 确认输入文本
2. 将输入文本标记为 `baseline`（基线版本）
3. 读取项目创作宪法：`system/constitution/MASTER.md`

---

## 📝 Step 1: 基线评估（Baseline Scoring）

**目标**：对原文进行双引擎AI味检测，建立改进基线。

### 引擎1：网文AI味检测（权重60%）

基于武行创作宪法，检测以下网文特有AI味：

| 类别 | 检测项 | 扣分 |
|------|--------|------|
| **神态微动作** | "嘴角勾起一抹弧度""瞳孔剧烈收缩""脸上带着笑意" | 每处 -5 |
| **滥用比喻** | "像一盆冰水""像一把精准的刻刀""宛如/仿佛" | 每处 -3 |
| **翻译腔** | "哦，我的老伙计""难道不…吗？" | 每处 -5 |
| **过度描写** | 单次环境描写超100字 | 每处 -3 |
| **词汇重复** | 同段出现两次相同形容词 | 每处 -2 |
| **陈词滥调** | "倒吸一口凉气""深邃的眼眸""死死的钉在" | 每处 -3 |
| **逻辑废话** | "他相信""不得不说""心中一动" | 每处 -2 |
| **无意义总结** | "这意味着…""可以预见…" | 每处 -3 |

### 引擎2：通用AI味检测（权重40%）

基于 Wikipedia 24种AI写作特征，检测以下通用AI味：

| # | 模式 | 检测信号 | 扣分 |
|---|------|---------|------|
| 1 | 夸大象征意义 | "标志着""是…的体现""凸显重要意义" | -3 |
| 2 | 过度强调知名度 | 堆砌媒体名/粉丝数但无引用 | -2 |
| 3 | -ing式肤浅分析 | "象征着…""反映出…" | -3 |
| 4 | 宣传广告式语言 | "充满活力的""令人叹为观止的" | -3 |
| 5 | 模糊归因 | "专家认为""行业报告显示" | -2 |
| 6 | 挑战与展望套路 | "尽管面临挑战…未来仍将…" | -3 |
| 7 | AI高频词 | "此外/至关重要/深入探讨/格局" | -2 |
| 8 | 回避"是"字 | 用"作为…""拥有…"代替判断句 | -2 |
| 9 | 否定式排比 | "不仅仅…而是…" | -2 |
| 10 | 三段式法则 | 强行分三块"三大优势/三个步骤" | -3 |
| 11 | 刻意换词 | 为避重复不断替换近义词 | -2 |
| 12 | 虚假范围 | "从X到Y"但X与Y无实际对比 | -1 |
| 13 | 破折号过度 | 用破折号强行插入解释 | -1 |
| 14 | 粗体过度 | 机械加粗关键词 | -1 |
| 15 | 内联标题列表 | 粗体标题+冒号开头 | -2 |
| 16 | 表情符号滥用 | 🚀💡✅等 | -1 |
| 17 | 协作交流痕迹 | "希望这对您有帮助！" | -3 |
| 18 | 截止日期免责 | "截至2023年""根据我的训练数据" | -3 |
| 19 | 谄媚语气 | "您说得完全正确！" | -3 |
| 20 | 填充短语 | "为了实现这一目标""值得注意的是" | -2 |
| 21 | 过度限定 | "可能""某种程度上""一般来讲" | -1 |
| 22 | 通用积极结论 | "未来一片光明""激动人心的时刻" | -3 |
| 23 | 过度压缩 | 丢失关键信息 | -2 |
| 24 | 模板残留 | 保留原文模板化表达 | -3 |

### 评分规则

```
基础分 = 100
网文AI味扣分 = Σ(各检测项扣分)
通用AI味扣分 = Σ(各检测项扣分)
最终分 = 100 - (网文扣分 × 0.6 + 通用扣分 × 0.4)
```

| 分数区间 | 等级 | 说明 |
|---------|------|------|
| 85-100 | A 优秀 | 自然流畅，几乎无AI味 |
| 70-84 | B 良好 | 有轻微AI味但不影响阅读 |
| 50-69 | C 及格 | AI味明显，需要优化 |
| <50 | F 不及格 | AI味严重，必须重写 |

**输出 baseline_score**，如果 ≥ 85 则直接通过，无需进入 Step 2。

---

## 🔄 Step 2: 生成挑战者（Generate Challengers）

**目标**：基于 baseline 生成多个不同风格的候选版本。

### Challenger-A：启发式自然版（heuristic-natural）

**策略**：打破AI的线性表达，注入"人味"

1. **注入内心吐槽和现代梗**：这是打破AI"一本正经"的最强武器
2. **打破均匀段落**：AI倾向生成长度一致的段落，用短句和对话加速
3. **删除填充短语**：去掉"值得注意的是""此外""需要指出的是"
4. **用行动展示心理**：删掉"心中一动""暗道"，改用行为暗示
5. **具体化抽象表达**：把"恐怖如斯"改为具体的恐惧表现

### Challenger-B：启发式平衡版（heuristic-balanced）

**策略**：在保留信息完整性的同时减少AI味

1. **保留核心剧情**：确保每个场景的关键信息不丢失
2. **简化过度修辞**：比喻每章不超过3个
3. **对话驱动**：用对话代替旁白解释
4. **短句优先**：超过20字的从句拆成短句
5. **相信读者**：删除"这意味着…"式的多余解释

### Challenger-C：规则驱动版（rule-driven）

**策略**：严格对照创作宪法逐条修复

1. 读取 `system/constitution/MASTER.md`
2. 逐条检查并修复违禁词/句式
3. 替换神态微动作为具体动作
4. 消除翻译腔
5. 控制环境描写在100字以内

---

## 📊 Step 3: 双引擎打分（Score & Compare）

**目标**：对所有候选（包括 baseline）进行统一评分。

**动作**：
1. 对 baseline、Challenger-A、Challenger-B、Challenger-C 全部执行 Step 1 的双引擎评分
2. 生成排名表：

```
| 版本 | 网文分 | 通用分 | 综合分 | 等级 |
|------|--------|--------|--------|------|
| baseline | 65 | 72 | 67.8 | C |
| Challenger-A | 82 | 78 | 80.4 | B |
| Challenger-B | 78 | 80 | 78.8 | B |
| Challenger-C | 85 | 75 | 81.0 | B |
```

3. 选择综合分最高的版本作为 `new_baseline`
4. 如果 `new_baseline` 综合分 ≥ 85，直接通过
5. 如果 < 85，进入 Step 4 修复轮

---

## 🔧 Step 4: 修复轮（Repair Round）

**目标**：基于当前最佳版本进行针对性修复，而非从头重写。

**动作**：
1. 分析 new_baseline 的扣分项，标记残留AI味标签：

| 失败标签 | 含义 |
|---------|------|
| `copied_baseline` | 直接复制了原文的AI味表达 |
| `template_carryover` | 模板化表达残留 |
| `bad_splice` | 修改和原文拼接不自然 |
| `placeholder_output` | 用占位符代替具体内容 |
| `over_compression` | 过度压缩导致信息丢失 |
| `lost_flavor` | 丢失了原文的独特文风 |

2. 针对每个失败标签生成修复版（Challenger-Repair）
3. 再次双引擎打分
4. 与 new_baseline 比较，更高则替换

---

## 🔁 Step 5: 迭代控制

**终止条件**（满足任一即停止）：

1. 综合分 ≥ 85
2. 已完成 3 轮修复
3. 连续 2 轮分数无提升（收敛）

**每轮结束后**：
- 展示当前最佳版本
- 展示与上一轮的变化点
- 等待用户确认是否继续

---

## 📋 Step 6: 输出最终版本

**输出内容**：

### 1. 最终优化文本
优化后的正文段落

### 2. 优化过程报告

```markdown
# 去AI味优化报告

## 基线评估
- 原文综合分: 67.8 (C)
- 主要问题: 神态微动作3处、滥用比喻2处、填充短语4处

## 优化轮次

### Round 1: 挑战者竞争
| 版本 | 综合分 | 变化 |
|------|--------|------|
| baseline | 67.8 | - |
| Challenger-A | 80.4 | +12.6 |
| Challenger-B | 78.8 | +11.0 |
| Challenger-C | 81.0 | +13.2 ✓ 最佳 |

### Round 2: 针对修复
- 标签: template_carryover(1处), bad_splice(1处)
| 版本 | 综合分 | 变化 |
|------|--------|------|
| new_baseline | 81.0 | - |
| Challenger-Repair | 87.2 | +6.2 ✓ 通过 |

## 最终结果
- 最终综合分: 87.2 (A)
- 提升: +19.4分
- 修改统计: 删除AI味表达12处, 重写3句, 新增具体动作描写2处
```

### 3. 变更对比
用 diff 格式展示原文→最终版的具体改动

---

## ⚠️ 注意事项

1. **保留作者风格**：优化不能抹杀原文的独特风格，只去除AI味
2. **信息完整性**：修复不能丢失关键剧情信息
3. **用户确认**：每轮结束后等待用户确认，不自动继续
4. **网文节奏**：不能为了去AI味而破坏网文的快节奏特性
5. **角色口癖**：保留已在 `SOLOENT.md §5` 注册的角色独特语言风格
